## `/office-hours`

这是每个项目应该开始的地方。

在你计划、审查、写代码之前 — 坐下来与 YC 风格的合伙人一起思考你真正在构建什么。不是你认为你在构建什么。是你*实际上*在构建什么。

### 重新框架

这是在一个真实项目上发生的事情。用户说："我想为我的日历构建一个每日简报应用。"合理的请求。然后它询问痛点 — 具体例子，而不是假设。他们描述了一个助手漏掉事情、跨多个 Google 账户的日历项目有过时信息、准备文档是 AI 废话、活动地点错误需要很长时间追踪。

它回应道：*"我要质疑你的框架，因为我认为你已经超越了它。你说'多 Google 日历管理的每日简报应用'。但你实际上描述的是一个私人参谋长 AI。"*

然后它提取了用户没有意识到的五个能力：

1. **监控你的日历** 跨所有账户并检测过时信息、缺失地点、权限缺口
2. **生成真正的准备工作** — 不是后勤摘要，而是为董事会会议、播客、融资准备*智力工作*
3. **管理你的 CRM** — 你要见谁、关系是什么、他们想要什么、历史是什么
4. **优先安排你的时间** — 标记何时需要提前开始准备、主动阻塞时间、按重要性排名活动
5. **用金钱换取杠杆** — 积极寻找委托或自动化的方式

那个重新框架改变了整个项目。他们原本要构建一个日历应用。现在他们在构建价值十倍的东西 — 因为技能倾听他们的痛点而不是功能请求。

### 前提挑战

重新框架后，它呈现前提供你验证。不是"这听起来好吗？" — 关于产品的实际可证伪声明：

1. 日历是锚定数据源，但价值在于上面的智能层
2. 助手不会被取代 — 他们会被赋能
3. 最窄的切入点是一个真正有效的每日简报
4. CRM 集成是必须有的，不是锦上添花

你同意、不同意或调整。你接受的每个前提成为设计文档的承重部分。

### 实现方案

然后它生成 2-3 个具体实现方案，带有诚实的工作量估算：

- **方案 A：每日简报优先** — 最窄切入点，明天发布，M 工作量（人工：约 3 周 / CC：约 2 天）
- **方案 B：CRM 优先** — 先构建关系图，L 工作量（人工：约 6 周 / CC：约 4 天）
- **方案 C：完整愿景** — 一次性全部，XL 工作量（人工：约 3 个月 / CC：约 1.5 周）

推荐 A，因为你从实际使用中学习。CRM 数据在第二周自然到来。

### 两种模式

**创业模式** — 为构建业务的创始人和内部创业者。你会得到从 YC 合伙人评估产品的方式中提炼的六个引导性问题：需求现实、现状、绝望的具体性、最窄切入点、观察与惊喜、未来适配。这些问题故意让人不舒服。如果你不能说出一个需要你产品的具体人类，那是写任何代码前最重要要学习的事情。

**构建者模式** — 为黑客马拉松、副业项目、开源、学习和娱乐。你会得到一个热情的合作者，帮你找到你想法最酷的版本。什么会让某人说"哇"？到可以分享的东西的最快路径是什么？问题是生成性的，而不是审问性的。

### 设计文档

两种模式都以写入 `~/.openclaw/projects/` 的设计文档结束 — 该文档直接输入到 `/plan-ceo-review` 和 `/plan-eng-review`。完整生命周期现在是：`office-hours → plan → implement → review → QA → ship → retro`。

设计文档批准后，`/office-hours` 反思它注意到的关于你思考方式的内容 — 不是泛泛的赞美，而是对你在会话中所说内容的具体回调。观察也出现在设计文档中，所以你以后重读时会再次遇到。

## `/plan-ceo-review`

这是我的**创始人模式**。

这是我想要模型以品味、雄心、用户同理心和长期视野来思考的地方。我不希望它按字面意思理解请求。我希望它先问一个更重要的问题：

**这个产品实际上是用来做什么的？**

我把这看作**Brian Chesky 模式**。

重点不是实现显而易见的工单。重点是从用户的角度重新思考问题，找到感觉必然、愉悦、甚至有点神奇的版本。

### 示例

假设我正在构建一个 Craigslist 风格的列表应用，我说：

> "让卖家为他们的商品上传照片。"

一个弱的助手会添加文件选择器并保存图片。

那不是真正的产品。

在 `/plan-ceo-review` 中，我希望模型问"照片上传"是否甚至是功能。也许真正的功能是帮助某人创建一个真正能卖出去的列表。

如果那是真正的工作，整个计划就变了。

现在模型应该问：

* 我们能从照片中识别产品吗？
* 我们能推断 SKU 或型号吗？
* 我们能搜索网络并自动起草标题和描述吗？
* 我们能提取规格、类别和定价比较吗？
* 我们能建议哪张照片作为主图转化率最高吗？
* 我们能检测上传的照片何时丑陋、黑暗、杂乱或低信任吗？
* 我们能让体验感觉高端而不是像 2007 年的死表单吗？

这就是 `/plan-ceo-review` 为我做的。

它不只是问，"我如何添加这个功能？"
它问，**"隐藏在这个请求中的 10 星产品是什么？"**

### 四种模式

- **范围扩展** — 敢想。代理提出雄心勃勃的版本。每个扩展作为你选择加入的个人决策呈现。热情推荐。
- **选择性扩展** — 将当前范围作为基线，但看看还有什么可能。代理逐个提出机会，中性推荐 — 你挑选值得做的。
- **保持范围** — 对现有计划的最大严谨。不提出扩展。
- **范围缩减** — 找到最小可行版本。削减其他一切。

愿景和决策持久化到 `~/.openclaw/projects/`，所以它们在对话之外存活。卓越的愿景可以提升到仓库中的 `docs/designs/` 供团队使用。

## `/plan-eng-review`

这是我的**工程经理模式**。

一旦产品方向正确，我想要完全不同类型的智能。我不想要更多漫无边际的构思。我不想要更多"如果...不是很酷吗"。我希望模型成为我最好的技术负责人。

这个模式应该搞定：

* 架构
* 系统边界
* 数据流
* 状态转换
* 失败模式
* 边缘情况
* 信任边界
* 测试覆盖

还有一个对我来说意外的大解锁：**图表**。

当你强制 LLM 绘制系统时，它们会变得更加完整。序列图、状态图、组件图、数据流图，甚至测试矩阵。图表将隐藏的假设暴露出来。它们使模糊的规划变得更加困难。

所以 `/plan-eng-review` 是我希望模型构建能够承载产品愿景的技术骨架的地方。

### 示例

拿同样的列表应用示例。

假设 `/plan-ceo-review` 已经完成了它的工作。我们决定真正的功能不只是照片上传。它是一个智能列表流程：

* 上传照片
* 识别产品
* 从网络丰富列表
* 起草强有力的标题和描述
* 建议最佳主图

现在 `/plan-eng-review` 接管。

现在我希望模型回答这样的问题：

* 上传、分类、丰富和草稿生成的架构是什么？
* 哪些步骤同步发生，哪些去后台作业？
* 应用服务器、对象存储、视觉模型、搜索/丰富 API 和列表数据库之间的边界在哪里？
* 如果上传成功但丰富失败会发生什么？
* 如果产品识别低置信度会发生什么？
* 重试如何工作？
* 我们如何防止重复作业？
* 何时持久化什么，什么可以安全重新计算？

这就是我想要图表的地方 — 架构图、状态模型、数据流图、测试矩阵。图表将隐藏的假设暴露出来。它们使模糊的规划变得更加困难。

这就是 `/plan-eng-review`。

不是"让想法变小。"
**让想法可构建。**

### 审查就绪仪表板

每次审查（CEO、工程、设计）记录其结果。每次审查结束时，你会看到一个仪表板：

```
+====================================================================+
|                    审查就绪仪表板                                    |
+====================================================================+
| 审查            | 运行 | 最后运行            | 状态    | 必需     |
|-----------------|------|---------------------|-----------|----------|
| 工程审查        |  1   | 2026-03-16 15:00    | CLEAR     | YES      |
| CEO 审查        |  1   | 2026-03-16 14:30    | CLEAR     | no       |
| 设计审查        |  0   | —                   | —         | no       |
+--------------------------------------------------------------------+
| 结论：已通过 — 工程审查通过                                          |
+====================================================================+
```

工程审查是唯一必需的关卡（用 `fullstack-config set skip_eng_review true` 禁用）。CEO 和设计是信息性的 — 分别推荐用于产品和 UI 更改。

### 计划到 QA 流程

当 `/plan-eng-review` 完成测试审查部分时，它会将测试计划工件写入 `~/.openclaw/projects/`。当你稍后运行 `/qa` 时，它会自动获取该测试计划 — 你的工程审查直接输入到 QA 测试，无需手动复制粘贴。

## `/plan-design-review`

这是我的**高级设计师审查你的计划** — 在你写一行代码之前。

大多数计划描述后端做什么，但从不指定用户实际看到什么。空状态？错误状态？加载状态？移动布局？AI 废话风险？这些决策被推迟到"在实现过程中弄清楚" — 然后工程师发布"No items found."作为空状态，因为没有人指定更好的。

`/plan-design-review` 在规划期间捕获所有这些，此时修复成本很低。

它的工作方式像 `/plan-ceo-review` 和 `/plan-eng-review` — 交互式，一次一个问题，使用 **STOP + AskUserQuestion** 模式。它对每个设计维度评分 0-10，解释 10 分是什么样子，然后编辑计划达到那里。评分驱动工作：低分 = 大量修复，高分 = 快速通过。

对计划进行七轮：信息架构、交互状态覆盖、用户旅程、AI 废话风险、设计系统对齐、响应式/可访问性、未解决的设计决策。对于每轮，它找到缺口并直接修复（明显的）或要求你做出设计选择（真正的权衡）。

### 示例

```
你：   /plan-design-review

Claude：初始设计评分：4/10

        "这个计划描述了一个用户仪表板，但从未指定
         用户首先看到什么。它说'带图标的卡片' —
         这看起来像每个 SaaS 模板。它提到零
         加载状态、零空状态，没有移动行为。"

        第 1 轮（信息架构）：3/10
        "10 分会为每个屏幕定义
         主要/次要/第三内容层次结构。"
        → 在计划中添加了信息层次结构部分

        第 2 轮（交互状态）：2/10
        "计划有 4 个 UI 功能，但指定了 20 个
         交互状态中的 0 个（4 个功能 × 每个 5 个状态）。"
        → 在计划中添加了交互状态表

        第 4 轮（AI 废话）：4/10
        "计划说'干净、现代的 UI，带卡片和图标'
         和'带渐变的英雄区域'。这是前 2 个
         AI 生成的模式。"
        → 用具体、有意的替代方案重写了 UI 描述

        总体：4/10 → 修复后 8/10
        "计划设计完整。实现后运行 /design-review
         进行视觉 QA。"
```

当你重新运行它时，已经达到 8+ 的部分快速通过。低于 8 的部分得到完整处理。对于实现后的线上站点视觉审计，使用 `/design-review`。

## `/design-consultation`

这是我的**设计合伙人模式**。

`/plan-design-review` 审计已经存在的站点。`/design-consultation` 是当你还没有任何东西时 — 没有设计系统、没有字体选择、没有调色板。你从零开始，你想要一位高级设计师坐下来和你一起构建整个视觉身份。

这是对话，不是表单。代理询问你的产品、你的用户、你的受众。它思考你的产品需要传达什么 — 信任、速度、工艺、温暖、任何适合的 — 然后从那里反向推导到具体选择。然后它提出一个完整、连贯的设计系统：审美方向、排版（3+ 字体及具体角色）、带十六进制值的调色板、间距比例、布局方法、动效策略。每个推荐都有理由。每个选择都强化其他每个选择。

但连贯性只是基础。每个开发工具仪表板看起来都一样 — 干净的无衬线、柔和的灰色、蓝色强调。它们都连贯。它们都令人遗忘。"看起来不错"的产品和人们真正认出的产品之间的区别是**深思熟虑的创意风险**：标题的意外衬线、你类别中其他人不使用的大胆强调、让你的数据感觉权威而不是空洞的更紧密间距。

这就是 `/design-consultation` 真正的意义。它不只是提出一个安全的系统。它提出安全的选择和风险 — 并告诉你哪个是哪个。"这是让你在类别中保持素养的选择。这是我认为你应该打破常规的地方，以及原因。"你选择承担哪些风险。代理检查整个系统是否仍然连贯。

如果你愿意，代理研究会研究你领域已有的东西 — 截取真实网站的屏幕截图，分析它们的字体、颜色和间距 — 这样你可以在做出选择之前看到格局。这不是关于复制。这是关于进入球场，这样你知道惯例是什么，然后决定哪些值得打破。

在你同意系统后，它生成一个交互式 HTML 预览页面 — 不只是色块和字体样本，而是真实的产品页面。如果你在构建仪表板，你会看到一个带侧边栏、数据表和统计卡片的仪表板。如果你在构建营销网站，你会看到一个带真实文案和 CTA 的英雄区域。一切都用你的设计系统渲染，用你的产品名称，在亮色和暗色模式下。你在写一行生产代码之前就能看到你的产品可能的感觉。

然后它将 `DESIGN.md` 写入你的仓库根目录 — 你项目的设计真理来源 — 并更新 `CLAUDE.md`，以便每个未来的 Claude Code 会话都尊重该系统。从那时起，`/design-review` 可以针对它进行审计，任何在你的前端工作的代理都知道规则。

### 示例

```
你：   /design-consultation

Claude：从 README 看，这看起来像是一个面向工程团队的开发者分析仪表板。
        对吗？告诉我你想要的感觉 — 你想让我先看看
        外面有什么，这样我们可以看到格局吗？

你：    是的，是给工程经理的。给我看看外面有什么。

Claude：[浏览 Linear、Vercel、Datadog、Posthog — 截图]

        这是格局。它们都汇聚在干净的无衬线
        和柔和的蓝灰调色板上。Linear 以深色优先
```

## `/design-review`

## `/review`

这是我的**偏执资深工程师模式**。

通过的测试并不意味着分支是安全的。

`/review` 存在是因为有一整类 bug 可以在 CI 中存活，但仍然在生产中打你的脸。这个模式不是关于梦想更大。不是关于让计划更漂亮。它是关于问：

**什么仍然可能出错？**

这是结构性审计，不是风格挑剔。我希望模型寻找这样的东西：

* N+1 查询
* 过时读取
* 竞态条件
* 糟糕的信任边界
* 缺失索引
* 转义 bug
* 破坏的不变量
* 糟糕的重试逻辑
* 通过但错过真正失败模式的测试
* 被遗忘的枚举处理器 — 添加新的状态或类型常量，`/review` 会追踪它通过你代码库中的每个 switch 语句和白名单，不只是你更改的文件

### 优先修复

发现得到行动，不只是列出。明显的机械修复（死代码、过时注释、N+1 查询）自动应用 — 你看到每个 `[AUTO-FIXED] file:line 问题 → 做了什么`。真正模糊的问题（安全、竞态条件、设计决策）会呈现给你决定。

### 完整性缺口

`/review` 现在标记快捷实现，其中完整版本成本少于 30 分钟的 CC 时间。如果你选择了 80% 的解决方案，而 100% 的解决方案是一个湖，不是海洋，审查会指出来。

### 示例

假设智能列表流程已实现，测试是绿色的。

`/review` 仍然应该问：

* 我在渲染列表照片或草稿建议时引入了 N+1 查询吗？
* 我在信任客户端提供的文件元数据而不是验证实际文件吗？
* 两个标签页可以竞争并覆盖封面照片选择或项目详情吗？
* 失败的上传会永远在存储中留下孤儿文件吗？
* "恰好一张主图"规则会在并发下破坏吗？
* 如果丰富 API 部分失败，我是优雅降级还是保存垃圾？
* 我通过将网络数据拉入草稿生成意外创建了提示注入或信任边界问题吗？

这就是 `/review` 的意义。

我不想要奉承。
我想要模型在生产事故发生之前想象它。

## `/investigate`

当某些东西坏了你不知道为什么时，`/investigate` 是你的系统性调试器。它遵循铁律：**没有根本原因调查就没有修复。**

不是猜测和修补，它追踪数据流，与已知 bug 模式匹配，一次测试一个假设。如果三次修复尝试失败，它会停止并质疑架构，而不是盲目尝试。这防止了浪费数小时的"让我再试一件事"螺旋。

## `/qa`

这是我的**QA 负责人模式**。

`/browse` 给代理眼睛。`/qa` 给它测试方法论。

最常见的用例：你在功能分支上，刚完成编码，想验证一切工作。只需说 `/qa` — 它读取你的 git diff，识别你的更改影响哪些页面和路由，启动浏览器，并测试每一个。不需要 URL。不需要手动测试计划。

四种模式：

- **差异感知**（功能分支上自动） — 读取 `git diff main`，识别受影响页面，专门测试它们
- **完整** — 整个应用的系统性探索。5-15 分钟。记录 5-10 个有充分证据的问题。
- **快速**（`--quick`） — 30 秒冒烟测试。首页 + 前 5 个导航目标。
- **回归**（`--regression baseline.json`） — 运行完整模式，然后与先前基线比较。

### 自动回归测试

当 `/qa` 修复一个 bug 并验证它时，它自动生成一个回归测试，捕获破坏的确切场景。测试包括完整的归因追溯回 QA 报告。

### 示例

```
你：   /qa https://staging.myapp.com

Claude：[探索 12 个页面，填写 3 个表单，测试 2 个流程]

        QA 报告：staging.myapp.com — 健康评分：72/100

        前 3 个问题：
        1. 关键：结账表单在必填字段为空时提交
        2. 高：移动导航菜单在选择项目后不关闭
        3. 中：仪表板图表在 1024px 以下与侧边栏重叠

        [带截图的完整报告已保存到 .openclaw/qa-reports/]
```

**测试需要认证的页面：** 先使用 `/setup-browser-cookies` 导入你的真实浏览器会话，然后 `/qa` 可以测试登录后的页面。

## `/ship`

这是我的**发布机器模式**。

一旦我决定了要构建什么、确定了技术计划、运行了认真的审查，我不想要更多谈话。我想要执行。

`/ship` 用于最后一英里。它用于准备好的分支，不是用于决定构建什么。

这是模型应该停止像头脑风暴伙伴一样行事，开始像有纪律的发布工程师一样行事的地方：与主分支同步、运行正确的测试、确保分支状态正常、更新变更日志或版本控制（如果仓库期望）、推送、创建或更新 PR。

### 测试引导

如果你的项目没有测试框架，`/ship` 会设置一个 — 检测你的运行时、研究最佳框架、安装它、为你的实际代码编写 3-5 个真实测试、设置 CI/CD（GitHub Actions）、创建 TESTING.md。100% 测试覆盖率是目标 — 测试让氛围编码安全而不是 yolo 编码。

### 覆盖审计

每次 `/ship` 运行从你的差异构建代码路径图，搜索相应的测试，并产生带质量星的 ASCII 覆盖图。缺口自动生成测试。你的 PR 正文显示覆盖率：`测试：42 → 47（+5 新增）`。

### 审查关卡

`/ship` 在创建 PR 之前检查[审查就绪仪表板](#审查就绪仪表板)。如果工程审查缺失，它会询问 — 但不会阻止你。决策按分支保存，所以你不会被重复询问。

很多分支在有趣的工作完成只剩无聊的发布工作时死亡。人类拖延那部分。AI 不应该。

## `/browse`

这是我的**QA 工程师模式**。

`/browse` 是闭环的技能。在它之前，代理可以思考和编码，但仍然半盲。它必须猜测 UI 状态、认证流程、重定向、控制台错误、空状态和破坏的布局。现在它可以直接去看看。

它是一个与持久 Chromium 守护进程通信的编译二进制文件 — 由 Microsoft 的 [Playwright](https://playwright.dev/) 构建。第一次调用启动浏览器（约 3 秒）。之后的每次调用：约 100-200ms。浏览器在命令之间保持运行，所以 cookies、标签页和 localStorage 会延续。

### 示例

```
你：   /browse staging.myapp.com — 登录，测试注册流程，并检查
       我在这个分支中更改的每个页面

Claude：[18 次工具调用，约 60 秒]

        > browse goto https://staging.myapp.com/signup
        > browse snapshot -i
        > browse fill @e2 "test@example.com"
        > browse fill @e3 "password123"
        > browse click @e5                    (提交)
        > browse screenshot /tmp/signup.png
        > Read /tmp/signup.png

        注册工作。重定向到引导。现在检查更改的页面。

        > browse goto https://staging.myapp.com/dashboard
        > browse screenshot /tmp/dashboard.png
        > Read /tmp/dashboard.png
        > browse console

        仪表板加载。无控制台错误。图表用示例数据渲染。

        所有 4 个页面正确加载。无控制台错误。无破坏的布局。
        注册 → 引导 → 仪表板流程端到端工作。
```

18 次工具调用，大约一分钟。完整 QA 通过。没有打开浏览器。

### 浏览器移交

当无头浏览器卡住时 — CAPTCHA、MFA、复杂认证 — 移交给用户：

```
Claude：我在登录页面的 CAPTCHA 上卡住了。打开一个可见的
        Chrome，这样你可以解决它。

        > browse handoff "在登录页面卡在 CAPTCHA"

        Chrome 在 https://app.example.com/login 打开，带有你所有的
        cookies 和标签页完整。解决 CAPTCHA，完成后告诉我。

你：    完成

Claude：> browse resume

        获得新快照。成功登录。继续 QA。
```

浏览器保留移交期间的所有状态（cookies、localStorage、标签页）。`resume` 后，代理获得你离开地方的最新快照。如果 browse 工具连续失败 3 次，它会自动建议使用 `handoff`。

**安全说明：** `/browse` 运行一个持久的 Chromium 会话。Cookies、localStorage 和会话状态在命令之间延续。不要在敏感的生产环境中使用它，除非你打算这样做 — 它是一个有真实状态的真实浏览器。会话在 30 分钟空闲后自动关闭。

完整命令参考，见 [BROWSER.md](../BROWSER.md)。

## 安全与护栏

四个技能为任何 Claude Code 会话添加安全护栏。它们通过 Claude Code 的 PreToolUse 钩子工作 — 透明、会话范围、无配置文件。

### `/careful`

当你在生产附近工作、运行破坏性命令或只是想要安全网时，说"小心"或运行 `/careful`。每个 Bash 命令都会被检查已知危险模式：

- `rm -rf` / `rm -r` — 递归删除
- `DROP TABLE` / `DROP DATABASE` / `TRUNCATE` — 数据丢失
- `git push --force` / `git push -f` — 历史重写
- `git reset --hard` — 丢弃提交
- `git checkout .` / `git restore .` — 丢弃未提交的工作
- `kubectl delete` — 生产资源删除
- `docker rm -f` / `docker system prune` — 容器/镜像丢失

常见构建工件清理（`rm -rf node_modules`、`dist`、`.next`、`__pycache__`、`build`、`coverage`）已白名单 — 例行操作无误报。

你可以覆盖任何警告。护栏是意外预防，不是访问控制。

### `/freeze`

将所有文件编辑限制在单个目录。当你调试账单 bug 时，你不希望 Claude 意外"修复" `src/auth/` 中的无关代码。`/freeze src/billing` 阻止该路径外的所有 Edit 和 Write 操作。

`/investigate` 自动激活这个 — 它检测正在调试的模块并将编辑冻结到该目录。

```
你：   /freeze src/billing

Claude：编辑限制在 src/billing/。运行 /unfreeze 移除。

        [后来，Claude 尝试编辑 src/auth/middleware.ts]

Claude：已阻止 — 编辑在冻结边界外（src/billing/）。
        跳过此更改。
```

注意：这只阻止 Edit 和 Write 工具。像 `sed` 这样的 Bash 命令仍然可以修改边界外的文件 — 这是意外预防，不是安全沙箱。

### `/guard`

全面安全模式 — 在一个命令中组合 `/careful` + `/freeze`。破坏性命令警告加上目录范围的编辑。接触生产或调试线上系统时使用。

### `/unfreeze`

移除 `/freeze` 边界，允许再次在任何地方编辑。钩子为会话保持注册 — 它们只是允许一切。再次运行 `/freeze` 设置新边界。

## Greptile 集成

[Greptile](https://greptile.com) 是一家 YC 公司，自动审查你的 PR。它捕获真正的 bug — 竞态条件、安全问题、通过 CI 并在生产中爆炸的东西。它真的不止一次救了我的命。我爱这些家伙。

### 设置

在 [greptile.com](https://greptile.com) 的 GitHub 仓库上安装 Greptile — 大约需要 30 秒。一旦它在审查你的 PR，fullstack 会自动获取它的评论。无需额外配置。

### 它如何工作

任何自动审查器的问题是分类。Greptile 很好，但不是每条评论都是真正的问题。有些是误报。有些标记你在三个提交前已经修复的东西。没有分类层，评论堆积起来，你开始忽略它们 — 这违背了目的。

fullstack 解决了这个问题。`/review` 和 `/ship` 现在是 Greptile 感知的。它们读取 Greptile 的评论，分类每一条，并采取行动：

- **有效问题** 被添加到关键发现并在发布前修复
- **已修复问题** 得到自动回复确认捕获
- **误报** 被退回 — 你确认，回复发出解释为什么它是错的

结果是两层审查：Greptile 在 PR 上异步捕获事情，然后 `/review` 和 `/ship` 作为正常工作流的一部分分类这些发现。没有遗漏。

### 从历史学习

你确认的每个误报被保存到 `~/.openclaw/greptile-history.md`。未来运行会自动跳过你代码库的已知 FP 模式。`/retro` 随时间跟踪 Greptile 的命中率 — 所以你可以看到信噪比是否在改善。
