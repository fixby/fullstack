# 预发布审查清单

## 说明

审查 `git diff origin/main` 输出中以下列出的问题。要具体 — 引用 `file:line` 并建议修复。跳过正常的内容。只标记真正的问题。

**两轮审查:**
- **第1轮 (关键):** 首先运行 SQL 与数据安全和 LLM 输出信任边界。最高严重性。
- **第2轮 (信息性):** 运行所有剩余类别。较低严重性但仍需处理。

所有发现通过修复优先审查处理:明显的机械修复自动应用,真正模糊的问题批量整理成单个用户问题。

**输出格式:**

```
Pre-Landing Review: N issues (X critical, Y informational)

**AUTO-FIXED:**
- [file:line] Problem → fix applied

**NEEDS INPUT:**
- [file:line] Problem description
  Recommended fix: suggested fix
```

如果未发现问题: `Pre-Landing Review: No issues found.`

要简洁。对于每个问题:一行描述问题,一行说明修复。无需前言、总结或"整体看起来不错"。

## 审查类别

### 第1轮 — 关键

#### SQL 与数据安全
- SQL 中的字符串插值(即使值是 `.to_i`/`.to_f` — 使用参数化查询(Rails: sanitize_sql_array/Arel; Node: prepared statements; Python: parameterized queries))
- TOCTOU 竞态条件:应该使用原子 `WHERE` + `update_all` 的检查-然后-设置模式
- 绕过模型验证直接写入数据库(Rails: update_column; Django: QuerySet.update(); Prisma: raw queries)
- N+1 查询:缺少预加载(Rails: .includes(); SQLAlchemy: joinedload(); Prisma: include)用于循环/视图中使用的关联

#### 竞态条件与并发
- 读取-检查-写入没有唯一性约束或捕获重复键错误并重试(例如,`where(hash:).first` 然后 `save!` 没有处理并发插入)
- 没有唯一数据库索引的 find-or-create — 并发调用可能创建重复项
- 不使用原子 `WHERE old_status = ? UPDATE SET new_status` 的状态转换 — 并发更新可能跳过或重复应用转换
- 不安全的 HTML 渲染(Rails: .html_safe/raw(); React: dangerouslySetInnerHTML; Vue: v-html; Django: |safe/mark_safe)用于用户控制的数据(XSS)

#### LLM 输出信任边界
- LLM 生成的值(电子邮件、URL、名称)写入数据库或传递给邮件程序而没有格式验证。在持久化之前添加轻量级防护(`EMAIL_REGEXP`、`URI.parse`、`.strip`)。
- 结构化工具输出(数组、哈希)在数据库写入之前没有类型/形状检查就被接受。

#### 枚举与值完整性
当差异引入新的枚举值、状态字符串、层级名称或类型常量时:
- **追踪每个消费者。** 阅读(不要只是 grep — 阅读)每个基于该值进行切换、过滤或显示的文件。如果任何消费者不处理新值,标记它。常见遗漏:向前端下拉菜单添加值但后端模型/计算方法不持久化它。
- **检查允许列表/过滤数组。** 搜索包含兄弟值的数组或 `%w[]` 列表(例如,如果向层级添加"revise",找到每个 `%w[quick lfg mega]` 并验证"revise"在需要的地方被包含)。
- **检查 `case`/`if-elsif` 链。** 如果现有代码基于枚举分支,新值是否会落入错误的默认值?
为此:使用 Grep 查找对兄弟值的所有引用(例如,grep "lfg" 或 "mega" 以找到所有层级消费者)。阅读每个匹配。此步骤需要阅读差异之外的代码。

### 第2轮 — 信息性

#### 条件副作用
- 基于条件分支但在一个分支上忘记应用副作用的代码路径。示例:项目升级为已验证但 URL 仅在次要条件为真时附加 — 另一个分支在没有 URL 的情况下升级,创建不一致的记录。
- 声称操作已发生但操作被有条件跳过的日志消息。日志应反映实际发生的情况。

#### 魔法数字与字符串耦合
- 在多个文件中使用的裸数字字面量 — 应该是记录在一起的命名常量
- 在其他地方用作查询过滤器的错误消息字符串(grep 该字符串 — 是否有内容匹配它?)

#### 死代码与一致性
- 已分配但从未读取的变量
- PR 标题与 VERSION/CHANGELOG 文件之间的版本不匹配
- 不准确描述更改的 CHANGELOG 条目(例如,"从 X 更改为 Y"当 X 从未存在时)
- 代码更改后描述旧行为的注释/文档字符串

#### LLM 提示问题
- 提示中的 0 索引列表(LLM 可靠地返回 1 索引)
- 提示文本列出与 `tool_classes`/`tools` 数组中实际连接的内容不匹配的可用工具/功能
- 在多个位置声明的单词/令牌限制可能会漂移

#### 测试缺口
- 断言类型/状态但不断言副作用的负向路径测试(URL 附加?字段填充?回调触发?)
- 对字符串内容的断言而不检查格式(例如,断言标题存在但不检查 URL 格式)
- 当代码路径应该明确不调用外部服务时缺少 `.expects(:something).never`
- 没有集成测试验证执行路径端到端工作的安全执行功能(阻止、速率限制、身份验证)

#### 完整性缺口
- 完整版本成本 <30 分钟 CC 时间的快捷实现(例如,部分枚举处理、不完整的错误路径、缺少易于添加的边缘情况)
- 仅提供人工团队工作量估算的选项 — 应该同时显示人工和 CC+fullstack 时间
- 添加缺失测试是"湖泊"而不是"海洋"的测试覆盖缺口(例如,缺少负向路径测试、缺少镜像快乐路径结构的边缘情况测试)
- 以 80-90% 实现的功能,而通过适度的额外代码可以实现 100%

#### 加密与熵
- 数据截断而不是哈希(最后 N 个字符而不是 SHA-256) — 熵较少,更容易冲突
- 对安全敏感值使用 `rand()` / `Random.rand` — 改用 `SecureRandom`
- 对秘密或令牌的非恒定时间比较(`==`) — 容易受到时序攻击

#### 时间窗口安全
- 假设"今天"覆盖 24 小时的日期键查找 — 上午 8 点 PT 的报告仅看到今天键下的午夜→上午 8 点
- 相关功能之间的时间窗口不匹配 — 一个使用小时桶,另一个对相同数据使用日键

#### 边界处的类型强制转换
- 值跨越 Ruby→JSON→JS 边界,类型可能改变(数字 vs 字符串) — 哈希/摘要输入必须规范化类型
- 序列化前不调用 `.to_s` 或等效项的哈希/摘要输入 — `{ cores: 8 }` vs `{ cores: "8" }` 产生不同的哈希

#### 视图/前端
- 部分中的内联 `<style>` 块(每次渲染重新解析)
- 视图中的 O(n*m) 查找(循环中的 `Array#find` 而不是 `index_by` 哈希)
- 对 DB 结果的 Ruby 端 `.select{}` 过滤,而可以是 `WHERE` 子句(除非有意避免前导通配符 `LIKE`)

#### 性能与包影响
- package.json 中已知较重的新 `dependencies` 条目:moment.js (→ date-fns, 330KB→22KB)、lodash 完整版(→ lodash-es 或按函数导入)、jquery、core-js 完整 polyfill
- 显著的 lockfile 增长(单个添加带来的许多新传递依赖)
- 没有 `loading="lazy"` 或显式 width/height 属性添加的图像(导致布局偏移 / CLS)
- 提交到仓库的大型静态资源(每个文件 >500KB)
- 没有 async/defer 的同步 `<script>` 标签
- 样式表中的 CSS `@import`(阻止并行加载 — 改用打包器导入)
- 依赖于另一个 fetch 结果的 fetch 的 `useEffect`(请求瀑布 — 合并或并行化)
- 可 tree-shake 库上的命名 → 默认导入切换(破坏 tree-shaking)
- ESM 代码库中的新 `require()` 调用

**不要标记:**
- devDependencies 添加(不影响生产包)
- 动态 `import()` 调用(代码分割 — 这些是好的)
- 小型实用程序添加(<5KB gzipped)
- 仅服务器端依赖项

## 严重性分类

```
CRITICAL (最高严重性):           INFORMATIONAL (较低严重性):
├─ SQL 与数据安全                ├─ 条件副作用
├─ 竞态条件与并发                ├─ 魔法数字与字符串耦合
├─ LLM 输出信任边界              ├─ 死代码与一致性
└─ 枚举与值完整性                ├─ LLM 提示问题
                                   ├─ 测试缺口
                                   ├─ 完整性缺口
                                   ├─ 加密与熵
                                   ├─ 时间窗口安全
                                   ├─ 边界处的类型强制转换
                                   ├─ 视图/前端
                                   └─ 性能与包影响

所有发现通过修复优先审查处理。严重性决定
呈现顺序和自动修复 vs 询问的分类 — 关键
发现倾向于询问(它们风险更高),信息性发现
倾向于自动修复(它们更机械化)。
```

## 修复优先启发式

此启发式被 `/review` 和 `/ship` 引用。它决定
代理是自动修复发现还是询问用户。

```
AUTO-FIX (代理不询问直接修复):     ASK (需要人工判断):
├─ 死代码 / 未使用的变量            ├─ 安全(身份验证、XSS、注入)
├─ N+1 查询(缺少预加载)            ├─ 竞态条件
├─ 与代码矛盾的陈旧注释            ├─ 设计决策
├─ 魔法数字 → 命名常量             ├─ 大型修复(>20 行)
├─ 缺少 LLM 输出验证               ├─ 枚举完整性
├─ 版本/路径不匹配                 ├─ 删除功能
├─ 已分配但从未读取的变量           └─ 任何改变用户可见
└─ 内联样式,O(n*m) 视图查找          行为的内容
```

**经验法则:** 如果修复是机械性的,高级工程师会
不经讨论就应用它,那就是自动修复。如果合理的工程师可能对
修复有不同意见,那就是询问。

**关键发现默认倾向于询问**(它们本质上风险更高)。
**信息性发现默认倾向于自动修复**(它们更机械化)。

## 抑制 — 不要标记这些

- 当冗余无害且有助于可读性时的"X 与 Y 冗余"(例如,`present?` 与 `length > 20` 冗余)
- "添加解释为什么选择此阈值/常量的注释" — 阈值在调优期间改变,注释会过时
- 当断言已经覆盖行为时"此断言可以更严格"
- 建议仅一致性的更改(将值包装在条件中以匹配另一个常量的保护方式)
- 当输入受限且 X 在实践中从未发生时"正则表达式不处理边缘情况 X"
- "测试同时执行多个保护" — 这没关系,测试不需要隔离每个保护
- 评估阈值更改(max_actionable、最低分数)— 这些是经验调优的并且不断变化
- 无害的空操作(例如,对从未在数组中的元素的 `.reject`)
- 你正在审查的差异中已解决的任何内容 — 在评论之前阅读完整差异
